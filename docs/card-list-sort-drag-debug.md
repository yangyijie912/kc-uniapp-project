# 卡片列表拖拽排序上拖死锁排障记录

这份文档单独记录这次顶边上拖“鬼打墙”的定位过程、根因和修复点。它不是设计总览，而是给后续复盘和排查留的可追溯记录。要看从手感改造到最终修复的完整时间线，去看 [./card-list-sort-drag-full-record.md](./card-list-sort-drag-full-record.md)。

## 1. 现象

拖拽排序在普通区域基本可用，但用户把卡片往上拖到列表顶边附近时，会出现这些体感：

- 卡片像停住了，不再往上推。
- 列表看起来在滚，但占位卡没有继续推进。
- 上面的卡片没有按预期下移，像原地打转。
- 反复拖几下才偶尔动一下，整体手感非常钝。

## 2. 先排除掉的方向

为了避免把无关问题混进来，先确认过这些点不是根因：

- 不是 `touchcancel` 的 H5 报错。
- 不是 placeholder 渲染丢失。
- 不是分页追加把 rect 彻底打乱。这里限定的是早期 H5/通用拖拽优化阶段里的“顶边上拖死锁”问题；后续 APP 长列表占位消失另有专项记录。
- 不是列表组件把 active card 直接从 DOM 里删掉导致命中错位。

这些问题前面已经各自处理过，但并不能解释“为什么一到上边缘就不往前推进”。

## 3. 真正的控制链

拖拽排序的关键链路是：

1. `handleSortHandleTouchMove` 只收集当前触点，塞进 `sortDragPendingTouchY`。
2. `runSortDragMoveFrame` 在动画帧里读取当前触点，更新拖拽代理卡的位置。
3. 如果贴近边缘，`autoScrollIfNeeded` 推动 `scrollTop`。
4. 再根据当前 touchY 通过 `getDragTargetByTouchY` 计算命中的锚点和 before/after。
5. 命中有效目标后，`moveCardLocally` 立刻重排 `cardList`，`sortInsertIndex` 通过 `currentMove` 和当前触点一起刷新占位。

## 4. 根因

问题不在“有没有滚”，而在“滚的时候有没有继续算命中并触发重排”。

当时的实现里，`runSortDragMoveFrame` 在检测到自动滚动后，会把本轮处理拆成两个节拍，极端情况下会让后半段命中和重排变得太保守。上拖到顶边时，用户其实最需要的是：

- 视口继续滚。
- 命中继续算。
- 列表继续换位。

如果自动滚动一触发就把后续重排链路切断，用户看到的就是“滚了，但没往前推”。

## 5. 修复点

这次修复做了三件事：

1. 自动滚动不再阻断同一帧里的命中计算和本地重排。
2. `getDragTargetByTouchY` 对上拖和下拖使用不同的命中边界，减少边缘附近反复踩线。
3. `sortInsertIndex` 在短暂算不到新目标时，沿用上一帧有效 move，让占位不要在边缘附近突然消失。

## 6. 为什么要这样改

这不是为了“让滚动更快”，而是为了把三个状态重新绑回同一条节奏里：

- 视口位置。
- 当前触点。
- 列表的局部顺序。

只要其中一个提前断开，拖拽就会显得僵硬。上拖时尤其明显，因为视口和列表顺序同时在动，任何一帧的空档都会被用户感知成“卡住了”。

## 7. 验证方式

可以按这个顺序验证：

1. 从列表中部开始拖拽。
2. 缓慢向上拖到第一屏顶部附近。
3. 观察占位卡是否会继续往上推进，而不是只看到视口在滚。
4. 再继续贴边拖几次，看上面的卡片是否能稳定下移。

如果这里仍然卡住，下一层就只需要看 `getDragTargetByTouchY` 的边界和 `currentMove` 的兜底是否一致，不需要重新怀疑整条拖拽链。
