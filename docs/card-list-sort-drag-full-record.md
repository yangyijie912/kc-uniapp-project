# 卡片列表拖拽排序完整记录

这份文档不是摘要，不是结论页，而是把这次拖拽排序从“手感改造”到“顶边上拖死锁修复”的全过程完整记下来。目的是可复盘、可追溯、可回看控制流，不让后面只剩“当时修过一个 bug”的模糊印象。

## 1. 起点：问题不是“拖不了”，而是“拖起来不跟手”

最初的用户感受很明确：

- 卡片拖拽太僵硬。
- 不是那种自然跟着手走的拖动感。
- 位置变化像“突然挪一下”，不是连续推进。
- 用户希望的是“真实拖动感”，不是浮层式的漂移效果。

所以这次改动从一开始就不是单纯修 bug，而是先把拖拽体验的表达方式改对。

## 2. 第一阶段：确认必须是“真实占位卡”，不是 guide line / overlay

一开始讨论过两类方案：

- 用 guide line 标线。
- 用悬浮 overlay 表达位置变化。

这两种都被否掉了，因为它们只是在视觉上“提示位置”，并不能让用户感到列表本身真的在腾挪。

最后确定的方案是：

- 保留拖拽中的代理卡。
- 在列表里插入一个真实的 placeholder 项。
- 让 transition-group 负责列表位移动画。

这个决定很关键，因为它直接决定了后面所有控制流都必须围绕“占位卡真实插入列表”来做，而不是靠额外画一条辅助线。

## 3. 第二阶段：列表渲染层落地

列表层的实现思路最终收敛成：

- `visibleCards` 负责把正在拖拽的 active card 从列表里过滤掉。
- `renderItems` 把 placeholder 按 `sortInsertIndex` 插回列表。
- `CardList.vue` 通过 `transition-group` 渲染卡片位移动画。
- 拖拽代理卡单独浮在列表外面，作为跟手反馈。

这个阶段里，最重要的不是样式，而是占位逻辑必须和真实列表顺序一致。否则用户看到的“卡片在移动”，和程序真正记录的“插入位置”会脱节。

## 4. 第三阶段：触摸事件和 H5 兼容问题

拖拽接到真实 touch 事件后，又出现了一个 H5 侧的明显问题：

- `touchcancel.prevent` 会在 H5 上报控制台错误。

处理方式是：

- 手柄上保留 `touchstart / touchmove / touchend` 的接管逻辑。
- `touchcancel` 去掉 `.prevent`，只保留 stop。
- 给拖拽手柄加 `touch-action: none`，避免浏览器抢走手势。

这一步虽然不是最终根因，但它保证了拖拽控制链不会一开始就被浏览器原生手势打断。

## 5. 第四阶段：rect 和数据同步

为了让命中计算不被 placeholder 和代理卡干扰，后续又做了几项关键修正：

- `refreshCardRects()` 只测量 `.card-item-sort-source`。
- active card 在拖拽期间从命中列表中排除。
- `watch(cardList)` 改成只在长度变化时刷新 rect，避免局部重排就把整个测量链反复打爆。
- `watch(scrollTop)` 在拖拽进行中只做缓存位移，不抢着重算预览。

这些修正的目标很直接：

- 让 rect 和列表顺序尽量保持同步。
- 不让 placeholder、代理卡、程序滚动彼此互相污染。

## 6. 第五阶段：真正开始出问题的是顶边上拖

当占位卡、代理卡、touch 事件都接上后，新的问题开始集中爆在一个场景里：

- 不能稳定往上拖。
- 一拖到列表顶边附近就像“停住了”。
- 有时看起来列表在滚，但上面的卡片没有真正让位。
- 有时又像同一位置来回闪，像“鬼打墙”。

这时已经不是展示层问题，而是控制流问题。

## 7. 第六阶段：最初的误判

一开始容易误以为是：

- 命中函数阈值太窄。
- placeholder 渲染不稳定。
- rect 刷新太慢。
- 页面分页导致列表前缀不准。

这些因素都确实存在过，但它们不是顶边死锁的最终根因。

更准确地说，它们会放大问题，但不是造成“上拖完全不往前推”的核心原因。

## 8. 第七阶段：真正的控制链

拖拽排序真正的链路是这样走的：

1. `handleSortHandleTouchMove` 只负责记录当前触点。
2. `runSortDragMoveFrame` 在动画帧里读取当前触点。
3. 先根据触点移动代理卡。
4. 再判断是否要自动滚动。
5. 再调用 `getDragTargetByTouchY` 计算当前命中的锚点和 before/after。
6. 命中有效目标时调用 `moveCardLocally`。
7. `moveCardLocally` 更新 `cardList`、`currentMove`，并触发 rect 刷新。
8. `sortInsertIndex` 再根据当前命中和上一帧有效 move 刷新占位。

这个链路里，只要有一环断掉，用户看到的拖拽都会变钝。

## 9. 第八阶段：最终根因

真正的问题出在：

- 自动滚动触发后，控制流把同一帧里的命中和重排拆断了。

更具体地说，当时的实现让 `runSortDragMoveFrame` 在边缘自动滚动后表现得过于保守，导致：

- 这一帧里只看到视口在变。
- 但 `getDragTargetByTouchY` 不一定继续推进到新的目标。
- `moveCardLocally` 也可能来不及执行。
- 占位卡在边缘附近没有及时换位，视觉上就像拖拽卡死。

也就是说，问题不是“滚没滚起来”，而是“滚起来以后有没有继续把列表换位这件事做完”。

## 10. 第九阶段：修复拆成三层

### 10.1 第一层：自动滚动不能早退打断同帧重排

先把最直接的断点修掉：

- 自动滚动继续保留。
- 但它不能提前中断同一帧里的命中和本地重排。

这样才能保证：视口在滚的同时，列表顺序也能继续推进。

### 10.2 第二层：给上拖和下拖不同的命中边界

只靠中线会让顶边附近容易“踩线”。

所以后续又加了方向感阈值：

- 上拖时命中边界更早切到上一张卡。
- 下拖时命中边界稍微往后推一点。
- 未知方向回退到中线。

这样做不是为了“更激进”，而是为了抵消边缘自动滚动带来的轻微滞后。

### 10.3 第三层：sortInsertIndex 的占位兜底

如果某一帧因为滚动回响或 rect 刷新暂时算不到新目标，不能立刻把 placeholder 清掉。

所以 sortInsertIndex 会在短暂失去新命中时沿用上一帧有效 move，保证占位卡不要在边缘附近突然消失。

## 11. 第十阶段：为什么这次不是“一个小 bug”

这次不是单点修复，而是整条链路一起暴露问题：

- 视觉层：要真实占位卡。
- 事件层：要正确接管 touch。
- 测量层：要让 rect 和列表同步。
- 滚动层：要区分原生滚动和程序滚动。
- 命中层：要按方向感判断上拖/下拖。
- 列表层：要在短暂失配时保住占位。

所以它的本质不是“把某个 if 改掉”，而是把一条拖拽控制链重新理顺。

## 12. 最终修复后的可验证现象

修复完成后，应该能看到这些变化：

- 往上拖到顶边时，不会只滚视口不换位。
- 占位卡会继续往上推进。
- 上面的卡会按预期下移。
- 上拖不再出现明显的“原地打转”感。

## 13. 这次过程里最容易踩的坑

这部分单独记下来，后面最有可能再次踩：

1. 只看滚动，不看同帧命中。
2. 只看命中，不看 rect 刷新。
3. 只看 placeholder，不看真实列表顺序。
4. 只看页面局部列表，不看分类全量保存。
5. 只看现象，不把控制流写清楚。
