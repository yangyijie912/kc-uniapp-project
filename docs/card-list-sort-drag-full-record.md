# 卡片列表拖拽排序完整记录

这份文档不是摘要，不是结论页，而是把这次拖拽排序从“手感改造”到“顶边上拖死锁修复”的全过程完整记下来。目的是可复盘、可追溯、可回看控制流，不让后面只剩“当时修过一个 bug”的模糊印象。

## 1. 起点：问题不是“拖不了”，而是“拖起来不跟手”

最初的用户感受很明确：

- 卡片拖拽太僵硬。
- 不是那种自然跟着手走的拖动感。
- 位置变化像“突然挪一下”，不是连续推进。
- 用户希望的是“真实拖动感”，不是浮层式的漂移效果。

所以这次改动从一开始就不是单纯修 bug，而是先把拖拽体验的表达方式改对。

## 2. 第一阶段：确认必须是“真实占位卡”，不是 guide line / overlay

一开始讨论过两类方案：

- 用 guide line 标线。
- 用悬浮 overlay 表达位置变化。

这两种都被否掉了，因为它们只是在视觉上“提示位置”，并不能让用户感到列表本身真的在腾挪。

最后确定的方案是：

- 保留拖拽中的代理卡。
- 在列表里插入一个真实的 placeholder 项。
- 让 transition-group 负责列表位移动画。

这个决定很关键，因为它直接决定了后面所有控制流都必须围绕“占位卡真实插入列表”来做，而不是靠额外画一条辅助线。

## 3. 第二阶段：列表渲染层落地

列表层的实现思路最终收敛成：

- `visibleCards` 负责把正在拖拽的 active card 从列表里过滤掉。
- `renderItems` 把 placeholder 按 `sortInsertIndex` 插回列表。
- `CardList.vue` 通过 `transition-group` 渲染卡片位移动画。
- 拖拽代理卡单独浮在列表外面，作为跟手反馈。

这个阶段里，最重要的不是样式，而是占位逻辑必须和真实列表顺序一致。否则用户看到的“卡片在移动”，和程序真正记录的“插入位置”会脱节。

## 4. 第三阶段：触摸事件和 H5 兼容问题

拖拽接到真实 touch 事件后，又出现了一个 H5 侧的明显问题：

- `touchcancel.prevent` 会在 H5 上报控制台错误。

处理方式是：

- 手柄上保留 `touchstart / touchmove / touchend` 的接管逻辑。
- `touchcancel` 去掉 `.prevent`，只保留 stop。
- 给拖拽手柄加 `touch-action: none`，避免浏览器抢走手势。

这一步虽然不是最终根因，但它保证了拖拽控制链不会一开始就被浏览器原生手势打断。

## 5. 第四阶段：rect 和数据同步

为了让命中计算不被 placeholder 和代理卡干扰，后续又做了几项关键修正：

- `refreshCardRects()` 只测量 `.card-item-sort-source`。
- active card 在拖拽期间从命中列表中排除。
- `watch(cardList)` 改成只在长度变化时刷新 rect，避免局部重排就把整个测量链反复打爆。
- `watch(scrollTop)` 在拖拽进行中只做缓存位移，不抢着重算预览。

这些修正的目标很直接：

- 让 rect 和列表顺序尽量保持同步。
- 不让 placeholder、代理卡、程序滚动彼此互相污染。

## 6. 第五阶段：真正开始出问题的是顶边上拖

当占位卡、代理卡、touch 事件都接上后，新的问题开始集中爆在一个场景里：

- 不能稳定往上拖。
- 一拖到列表顶边附近就像“停住了”。
- 有时看起来列表在滚，但上面的卡片没有真正让位。
- 有时又像同一位置来回闪，像“鬼打墙”。

这时已经不是展示层问题，而是控制流问题。

## 7. 第六阶段：最初的误判

一开始容易误以为是：

- 命中函数阈值太窄。
- placeholder 渲染不稳定。
- rect 刷新太慢。
- 页面分页导致列表前缀不准。

这些因素都确实存在过，但它们不是顶边死锁的最终根因。

更准确地说，它们会放大问题，但不是造成“上拖完全不往前推”的核心原因。

## 8. 第七阶段：真正的控制链

拖拽排序真正的链路是这样走的：

1. `handleSortHandleTouchMove` 只负责记录当前触点。
2. `runSortDragMoveFrame` 在动画帧里读取当前触点。
3. 先根据触点移动代理卡。
4. 再判断是否要自动滚动。
5. 再调用 `getDragTargetByTouchY` 计算当前命中的锚点和 before/after。
6. 命中有效目标时调用 `moveCardLocally`。
7. `moveCardLocally` 更新 `cardList`、`currentMove`，并触发 rect 刷新。
8. `sortInsertIndex` 再根据当前命中和上一帧有效 move 刷新占位。

这个链路里，只要有一环断掉，用户看到的拖拽都会变钝。

## 9. 第八阶段：最终根因

真正的问题出在：

- 自动滚动触发后，控制流把同一帧里的命中和重排拆断了。

更具体地说，当时的实现让 `runSortDragMoveFrame` 在边缘自动滚动后表现得过于保守，导致：

- 这一帧里只看到视口在变。
- 但 `getDragTargetByTouchY` 不一定继续推进到新的目标。
- `moveCardLocally` 也可能来不及执行。
- 占位卡在边缘附近没有及时换位，视觉上就像拖拽卡死。

也就是说，问题不是“滚没滚起来”，而是“滚起来以后有没有继续把列表换位这件事做完”。

## 10. 第九阶段：修复拆成三层

### 10.1 第一层：自动滚动不能早退打断同帧重排

先把最直接的断点修掉：

- 自动滚动继续保留。
- 但它不能提前中断同一帧里的命中和本地重排。

这样才能保证：视口在滚的同时，列表顺序也能继续推进。

### 10.2 第二层：给上拖和下拖不同的命中边界

只靠中线会让顶边附近容易“踩线”。

所以后续又加了方向感阈值：

- 上拖时命中边界更早切到上一张卡。
- 下拖时命中边界稍微往后推一点。
- 未知方向回退到中线。

这样做不是为了“更激进”，而是为了抵消边缘自动滚动带来的轻微滞后。

### 10.3 第三层：sortInsertIndex 的占位兜底

如果某一帧因为滚动回响或 rect 刷新暂时算不到新目标，不能立刻把 placeholder 清掉。

所以 sortInsertIndex 会在短暂失去新命中时沿用上一帧有效 move，保证占位卡不要在边缘附近突然消失。

## 11. 第十阶段：为什么这次不是“一个小 bug”

这次不是单点修复，而是整条链路一起暴露问题：

- 视觉层：要真实占位卡。
- 事件层：要正确接管 touch。
- 测量层：要让 rect 和列表同步。
- 滚动层：要区分原生滚动和程序滚动。
- 命中层：要按方向感判断上拖/下拖。
- 列表层：要在短暂失配时保住占位。

所以它的本质不是“把某个 if 改掉”，而是把一条拖拽控制链重新理顺。

## 12. 最终修复后的可验证现象

修复完成后，应该能看到这些变化：

- 往上拖到顶边时，不会只滚视口不换位。
- 占位卡会继续往上推进。
- 上面的卡会按预期下移。
- 上拖不再出现明显的“原地打转”感。

## 13. App 端补充问题：代理未就绪时的空白态

在 H5 上，拖拽代理通常能很快拿到尺寸并显示出来；但 App 端在拖拽刚开始的那一帧，有时会出现代理样式还没准备好、但 active card 已经被隐藏的空档。

如果这时直接按 sortActiveId 把原卡片从列表里过滤掉，页面就会短暂只剩下空白感。这个问题不是主拖拽逻辑坏了，而是“隐藏 active card”的时机太早。

后续在列表层补了一个代理就绪判断：只有拖拽代理已经拿到有效样式时，才把 active card 从可见列表中移走。这样 App 端就算代理晚一拍，也不会先进入空白态。

补充一点：如果 App 端仍出现整组短暂空白，优先怀疑的是列表更新容器本身，而不是卡片数据真被清空。原生端对 `transition-group` 的更新过程比 H5 更脆弱，所以列表层已经准备了 plain `view + v-for` 的降级路径，避免动画容器在 App 里把同步更新表现成“全没了”。

## 14. 这次过程里最容易踩的坑

这部分单独记下来，后面最有可能再次踩：

1. 只看滚动，不看同帧命中。
2. 只看命中，不看 rect 刷新。
3. 只看 placeholder，不看真实列表顺序。
4. 只看页面局部列表，不看分类全量保存。
5. 只看现象，不把控制流写清楚。

## 16. 第十一阶段：APP 端触摸链路重构

顶边上拖问题收口后，APP 真机仍暴露出更底层的问题：浮起后卡片不跟手、不能边拖边滚、占位卡位置跳动。

这阶段确认了一个关键差异：H5 可以把拖拽源卡片隐藏或从列表里移走，APP 端不行。APP 端后续 `touchmove` 很依赖最初按住的源节点仍然存在。如果源节点被 `display: none`、被条件渲染移除，或者被 `transition-group` 替换，拖拽链路就可能断掉。

最终 APP 端改成：

- 真实源卡片保留在 DOM 中。
- 源卡片透明并移出文档流，避免占据列表空间。
- 视觉上的浮起卡片交给 `CardDragProxy.vue`。
- APP 列表用普通 `view + v-for`，不走 `transition-group`。

这个阶段修复了“浮起来就不动”和“双卡显示”的主要问题。双卡问题的根因是平台判断不稳，源卡片没有正确进入透明脱流态，导致真实卡片和代理卡同时显示。

## 17. 第十二阶段：边缘滚动从混用 scrollTop 改成 scrollTopTarget

APP 端边拖边滚最初失败，是因为同一个 `scrollTop` 同时承担了两个角色：

- `@scroll` 回来的真实位置。
- 程序自动滚动要写入的目标位置。

在 APP 真机里，这会互相覆盖：程序刚写了一个目标，原生滚动回调又把旧位置写回来，结果表现成不滚、跳滚、滚到最后或上滚鬼打墙。

后续拆成：

- `scrollTop`：只记录真实滚动。
- `scrollTopTarget`：只绑定给 `scroll-view` 作为程序滚动目标。

这一步让“拖拽优先，到边缘后程序滚动”重新成立。之后边缘滚动参数经历了几轮调试：过宽会一往下拖就触发，过窄又滚得慢。最终保留的是中档速度和较克制的边缘热区。

## 18. 第十三阶段：分页追加导致占位只认第一页

用户后来发现：如果数据全部加载完成，占位卡消失问题会明显减少；如果拖拽中继续触发第二页、第三页加载，占位卡会停在第一页或旧页尾部，被滚出屏幕。

这个线索确认问题不在保存模型。保存模型本来就是 `Move`，允许页面只拿局部列表，松手时交给服务层按分类全量顺序回放。真正的问题在拖拽命中系统：

1. 分页追加后，`cardList` 变长。
2. DOM 节点集合变长。
3. 如果旧 rect 继续参与命中，占位就不会推进到新页。
4. 如果按 `cardRects[index]` 对齐 `cardList[index]`，APP 端很容易因为透明源节点、占位卡、本地重排而错位。

中途曾尝试拖拽中禁止分页追加，但这个方案被撤回，因为它违背“边拖边滑，滑到底继续加载”的原设计。

最终修复：

- 卡片节点增加 `data-card-id`。
- rect 采集改成 `.fields({ id, dataset, rect, size })`。
- 命中时从 rect 自己的 `dataset.cardId` 或 id 反推卡片 id。
- 有效 rect 按屏幕 `top` 排序。
- `watch(cardList)` 在拖拽中发现长度变化后，强制刷新 rect，而不是只排队到下一帧。
- 分页追加后用当前触点重新刷新占位。

## 19. 第十四阶段：占位状态从 computed 改成显式 ref

长列表和快速往返拖暴露了另一个问题：有些路径更新了 `sortDragPreviewMove`，但没有同步更新占位下标。结果就是：

- 占位卡突然消失。
- 占位卡停在旧位置。
- 往返拖时，视觉代理卡和占位下标不同步。

后续把 `sortInsertIndex` 从 computed 改成显式 `ref`，并把所有占位更新收口到 `applySortDragTarget`：

- 普通 touchmove 走它。
- 动画帧内本地重排走它。
- 自动滚动循环推进视口后也走它。

这样 `sortDragPreviewMove` 和 `sortInsertIndex` 必须一起更新，不能再出现 move 变了但占位下标没变。

## 20. 第十五阶段：尾卡落点为什么要特殊处理

最后一个顽固问题是尾卡落点。

现象：

- 拖住最后一张卡继续往下拖，占位卡不出现。
- 先上拖再下拖，占位卡可能落到倒数第二张附近。

这里的关键是不对称：首卡只要进入第一张非源卡的上半区，就能表达 `before`；尾卡则往往要进入最后卡的下半区甚至越过底边，才能表达 `after`。APP 底部固定操作栏会盖住最后卡片的自然 after 区域；真机调试期间打开的调试面板会进一步压缩这块区域，但正常构建里调试面板默认关闭。

另外，拖拽源卡片本身如果就是最后一张，那么命中列表会排除源卡。此时如果尾部 fallback 取最后一个可见 rect，很容易变成倒数第二张。

最终修复：

- 下拖时，命中点进入最后一个可见锚点的下半区，就直接视为插入当前已加载列表尾部。
- 尾部 fallback 使用 `cardList` 当前已加载顺序里的最后一张非源卡，而不是最后一个 rect。
- 占位下标仍然按“排除拖拽源卡后的列表”计算；APP 渲染层再根据源卡片保留情况修正 DOM 插入位置。

## 21. 当前最终链路

当前 APP/H5 共用的核心链路是：

1. 手柄 `touchstart` 开始拖拽，采集 rect。
2. APP 保留源节点，H5 可隐藏源节点。
3. `CardDragProxy` 渲染固定定位的视觉代理卡。
4. `touchmove` 更新代理卡位置和命中点。
5. 边缘区域推进 `scrollTopTarget`。
6. 滚动变化先平移缓存 rect，再安排真实 rect 重采。
7. 分页追加后强制重采 rect。
8. `applySortDragTarget` 同步更新预览 move 和占位下标。
9. 本地重排只负责即时反馈。
10. `touchend` 使用最终 move 交给 `updateCardOrderInCategory`，由服务层基于分类全量顺序回放保存。

这条链路的关键边界仍然是：页面只代表当前已加载列表，不能把页面列表当成最终顺序全量；服务层才负责全量一致性。
