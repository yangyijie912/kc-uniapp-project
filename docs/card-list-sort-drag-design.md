# 卡片列表排序与拖拽设计复盘

## 1. 背景

这轮改动的目标，不是单纯“给列表加一个排序下拉框”，而是把卡片列表页里和顺序有关的能力补完整：

- 接入统一的排序参数链路，让列表查询真正按选中的排序方式返回。
- 在分类列表中支持拖拽排序，并把结果持久化到本地 storage。
- 处理搜索、筛选、多选、拖拽几种交互模式之间的互斥关系，避免状态残留。
- 在 uni-app 的 scroll-view 场景下，尽量保证“手柄优先”和“边拖边滚”同时成立。

这次实现覆盖的主要文件：

- src/pages/cardList/index.vue
- src/composables/useCardSortDrag.ts
- src/composables/useCardListView.ts
- src/services/cardService.ts
- src/utils/cardSort.ts
- src/types/card.ts

## 2. 问题定义

改造前有几类核心问题：

### 2.1 排序参数虽然加了，但查询没有真正落到服务层

列表页最初把排序选项接到了页面状态上，但服务层查询里把 cardSortConfig 混进了普通字段过滤，结果导致卡片列表被错误过滤，出现“卡片一张都没出来”的问题。

根因不是 UI 没接上，而是查询参数分层不清：

- `keyword` 属于模糊查询。
- `status`、`categoryId` 属于精确过滤。
- `cardSortConfig` 属于排序控制。

这三类参数必须在服务层分开处理。

### 2.2 拖拽排序和分页列表天然冲突

页面层的 `cardList` 不是分类下的全量数据，而是“当前已加载前 N 条”的前缀列表。这个设计本来是为了无限滚动，但它直接带来一个限制：

- 页面上看到的顺序，只能代表当前已加载片段。
- 真正持久化排序时，不能直接把当前 `cardList` 当成完整基线覆盖保存。

如果直接以页面当前顺序写回 storage，会把未加载的同分类卡片顺序弄乱。

### 2.3 多种交互模式会互相打架

列表页同时存在这些状态：

- 浏览
- 长按进入多选
- 搜索/筛选
- 拖拽排序

如果继续用多个布尔值各自管理，会出现这些典型问题：

- 正在多选时切排序，状态没清干净。
- 搜索结果页还能进入拖拽排序。
- 长按刚结束，点击事件又把页面跳进详情。

### 2.4 uni-app 里拖拽和滚动容易互相抢控制权

在 scroll-view 中做手势拖拽时，实际会同时遇到三套行为：

- 手柄本身的 touch 事件。
- scroll-view 原生滚动。
- 程序为了“贴边自动滚动”而主动改写的 `scrollTop`。

如果不区分“用户滚动”和“程序滚动”，最终就会出现：

- 拖着拖着不滚。
- 一滚就丢命中。
- 自动滚动和原生滚动来回打架。

## 3. 设计目标

这次方案收敛到四个明确目标：

1. 排序查询链路统一，页面、composable、服务层对排序含义一致。
2. 拖拽排序只在“分类列表且非搜索结果”时开放，不做跨分类排序。
3. 拖拽过程要有即时视觉反馈，但持久化必须基于分类全量顺序。
4. 页面只保留一个交互模式源，其他行为都围绕它做约束。

## 4. 方案选择与取舍

### 4.1 交互模式：从多个布尔值改为单一 interactionMode

最终采用：

- `browse`
- `select`
- `sort`

而不是继续保留多个布尔状态并行。

原因：

- 浏览、多选、拖拽在产品语义上就是互斥态，不应该允许并存。
- 单一模式源更容易做切换时的清理，例如退出多选时清空 selectedCards，退出拖拽时清空拖拽态。
- 页面上所有“先退出其他模式再做下一步”的逻辑都能统一收口。

对应实现：

- 页面层保留 `interactionMode` 作为唯一模式源。
- `useCardSortDrag` 基于这个模式源计算 `isSortMode`、模式标题、模式说明和排序按钮状态。

### 4.2 拖拽排序的适用范围：只支持分类列表，不支持全部列表和搜索结果

这是刻意限制，不是功能缺失。

原因：

- 全部列表本身是跨分类聚合结果，而自定义排序是“分类内顺序”。
- 搜索结果页是按关键字筛出来的临时集合，不适合作为排序落库视图。
- 如果在这些场景里开放拖拽，用户看到的是局部集合，但要保存的是分类内全量顺序，心智模型会错位。

因此最终规则是：

- 只有 `categoryId` 存在且不处于搜索结果模式时，允许进入排序模式。
- 进入失败时直接给出明确原因提示。

### 4.3 持久化模型：最终采用 Move，而不是直接保存最终顺序数组

这轮中途讨论过两种保存策略。

方案 A：页面拖完后直接把最终顺序数组整体保存。

放弃原因：

- 页面里的 `cardList` 只是已加载片段，不是分类全量。
- 直接保存最终顺序数组，必须拿到分类全量且保证顺序绝对正确，否则会覆盖出错。

方案 B：拖拽过程中只记录最后一次有效移动 `Move`。

最终采用的数据结构：

```ts
type Move = {
  movedId: string;
  anchorId: string;
  position: 'before' | 'after';
};
```

采用原因：

- 页面本地重排只负责即时反馈。
- 松手时只需要把“最后一次有效移动”交给服务层。
- 服务层可以基于分类全量顺序重新计算并落库，不依赖页面是否只加载了前 N 条。

这个取舍的本质是：

- 页面负责交互反馈。
- 服务层负责基于完整数据做最终一致性保存。

### 4.4 落库策略：重写整个分类的 sort，而不是只改单张卡片的 sort

最终在 `updateCardOrderInCategory` 里采用的做法是：

1. 读取 storage 全量卡片。
2. 过滤出当前分类下的所有卡片，并按当前自定义顺序排序。
3. 应用这次 `Move`。
4. 按新的分类内顺序重写每张卡片的 `sort`，使用固定步长 `SORT_STEP = 1000`。
5. 写回整份卡片列表。

没有采用“只更新被拖动卡片的 sort 值”这类局部写法，原因是：

- 局部插值方案看起来省写入，但边界更多。
- 多次拖拽后容易出现 sort 过密、比较不稳定或需要额外整理的问题。
- 当前数据规模下，直接重写整个分类顺序更直观，也更容易复盘和排错。

## 5. 最终实现结构

### 5.1 页面层：负责参数、模式切换和普通列表行为

`src/pages/cardList/index.vue` 负责：

- 解析页面参数。
- 统一组装查询参数 `buildQuery(page)`。
- 承接搜索、状态筛选、排序切换、分页加载。
- 维护多选相关状态。
- 把拖拽排序相关逻辑委托给 `useCardSortDrag`。

页面层保留的关键原则：

- 所有重新加载都走 `loadCurrentPages()`。
- 搜索、筛选、排序切换前先 `resetInteractionModes()`。
- 页面只关心“当前是什么模式”，不直接处理拖拽命中和自动滚动细节。

### 5.2 useCardListView：负责“当前已加载前缀列表”

`src/composables/useCardListView.ts` 的设计没有改成真正的分页页码追加，而是继续维持当前项目已经在用的“请求前 N 条”策略：

- 第 1 页取前 10 条。
- 第 2 页取前 20 条。
- 第 3 页取前 30 条。

这能保持当前列表视图和返回页面后的数据体验一致，但也带来一个明确前提：

- `cardList` 只是已加载前缀，不是全量基线。

所以排序保存不能依赖它。

### 5.3 useCardSortDrag：负责拖拽会话、命中计算和滚动协调

`src/composables/useCardSortDrag.ts` 承担的职责有：

- 排序模式是否允许进入。
- 排序模式的 UI 文案。
- 当前拖拽态卡片的激活状态。
- 卡片 rect 和 scroll-view rect 的采集。
- 手指命中的锚点计算。
- 本地列表即时重排。
- 贴边自动滚动。
- touchend 时把最后一次 move 交给服务层保存。

拖拽链路的实际顺序是：

1. `touchstart` 时记录当前拖拽卡片，并采集一次 rect。
2. `touchmove` 时先判断是否需要自动滚动。
3. 再根据最新 rect 计算当前命中的锚点卡片和落点位置。
4. 命中有效目标后，本地调整 `cardList` 顺序并刷新 rect。
5. `touchend` 时清理临时状态，并将最后一次 `Move` 提交给服务层。

### 5.4 cardService：负责最终一致性保存

`src/services/cardService.ts` 在这轮里新增了两个关键点：

- `getCards` 中把 `cardSortConfig` 从过滤字段里拆出来，单独作为排序配置处理。
- `updateCardOrderInCategory(categoryId, move)` 负责按分类全量顺序应用拖拽结果并保存。

这里的责任边界很重要：

- 页面不做最终顺序真值。
- 服务层才是顺序持久化的唯一落点。

## 6. 关键实现细节

### 6.1 为什么进入排序模式时强制切到 customSort

如果当前列表还停留在“按创建时间”或“按更新时间”排序，用户看到的顺序就不是自定义顺序。此时允许拖拽，会出现明显的认知冲突：

- 拖的是一套顺序。
- 保存的却是另一套自定义顺序。

因此进入排序模式时，会先把排序项切到 `customSort`，再展示拖拽手柄。

退出排序模式时不自动切回原排序，原因也明确：

- 用户刚刚排完序，立即切回别的排序会让列表看起来“排序没生效”。

### 6.2 为什么本地重排后还要频繁 refreshCardRects

因为每次局部重排或滚动后，屏幕上每张卡片的位置都变了。命中逻辑依赖 rect，如果不重新采集，下一次 touchmove 很容易按旧坐标继续算，表现出来就是：

- 命中错卡。
- 拖动时顺序跳变。
- 到边缘后拖拽目标突然失准。

### 6.3 如何区分原生滚动和程序自动滚动

这部分是这轮里最容易踩坑的实现点。

当前方案使用三个时间状态协同：

- `lastAutoScrollAt`：最近一次程序自动滚动的时间。
- `lastNativeScrollAt`：最近一次被判定为原生滚动的时间。
- `programmaticScrollUntil`：代码修改 `scrollTop` 后的回响保护窗口。

工作方式：

1. 程序推进 `scrollTop` 时，写入 `programmaticScrollUntil = now + autoScrollCooldown`。
2. `watch(scrollTop)` 收到变化时，如果当前时间还没超过这个值，就只刷新 rect，不记成原生滚动。
3. 超过这个窗口后收到的滚动变化，才判定为用户手势滚动，并更新 `lastNativeScrollAt`。
4. `autoScrollIfNeeded` 内再根据 `nativeScrollQuietWindow` 短暂停止程序自动滚动。

这样做的目的是让两种滚动在时间上错峰，而不是同时抢控制权。

## 7. 踩坑记录

### 7.1 排序参数误参与精确过滤，直接导致空列表

这是最先暴露出来的问题。现象是“排序接上后卡片一张都没出来”，根因是：

- `getCards(params)` 里把 `cardSortConfig` 当普通字段参与了卡片过滤。

修复方式：

- 在服务层先把 `cardSortConfig` 解构出去，只对剩余字段做精确匹配。

### 7.2 把页面局部列表误当成分类全量顺序

这是排序持久化里最大的结构性风险。

最开始如果沿着“页面已经排好了，直接保存页面顺序”这个思路走，会天然踩到分页前缀列表的坑。后面明确改成 `Move + 服务层全量回放`，就是为了解这个问题。

### 7.3 文案和真实行为不一致

中途出现过“完成排序”这类文案，但实际行为并不是“点击后统一提交排序”，而是：

- 拖拽时本地跟手。
- 松手时自动保存。
- 按钮本质是进入/退出排序模式。

所以后来把文案改成“拖拽排序 / 退出拖拽”，避免 UI 暗示错误的交互模型。

### 7.4 粗暴禁用滚动并不能解决拖拽问题

一开始很容易走到一个表面正确的方案：

- 进入拖拽后把原生滚动关掉。

但这样会直接带来另一个问题：

- 用户拖到长列表边缘时，根本没法继续往下拖。

最终还是回到“保留原生滚动，但用静默窗口协调程序滚动”的方案。

### 7.5 手柄事件必须优先接管

如果不在手柄上使用 `touchstart/touchmove/touchend` 的 `stop + prevent`，拖拽过程中父级卡片的点击、长按、多选逻辑就很容易混进来，导致：

- 拖着拖着触发多选。
- 松手后误进详情。

所以最终明确采用“点中手柄，优先手柄”的事件策略。

## 8. 这次方案的边界

当前实现明确不处理这些能力：

- 跨分类拖拽排序。
- 搜索结果集合内拖拽排序。
- 全部列表视图下的拖拽排序。
- 基于页面当前局部列表直接覆盖最终顺序。

这些不是遗漏，而是当前数据模型和交互模型下的主动约束。

## 9. 验证情况

这轮改动完成后，已做过的校验包括：

- 相关核心文件的文件级错误检查通过。
- 关键控制链已补注释，便于后续继续排查和迭代。

当前仍保留的验证边界：

- 项目全量 `type-check` 仍受 tsconfig 中 `ignoreDeprecations: "6.0"` 影响，现阶段无法作为有效收口校验。
- 拖拽手感参数（如边缘触发距离、滚动步长、静默窗口）仍然是体验参数，需要继续在真机和长列表场景里微调。

## 10. 后续可继续演进的方向

如果后续要继续做这块，建议按下面的顺序推进：

1. 真机验证并微调拖拽边缘滚动参数。
2. 给拖拽排序补行为级测试或最小可复现用例说明。
3. 如果未来要支持更大数据量，再评估是否需要从“重写整个分类 sort”切到更复杂的局部插值策略。
4. 如果未来要支持跨分类排序，先重做数据模型和交互边界，不建议在现有语义上硬扩。
