# 卡片列表排序与拖拽设计复盘

## 1. 背景

这轮改动的目标，不是单纯“给列表加一个排序下拉框”，而是把卡片列表页里和顺序有关的能力补完整：

- 接入统一的排序参数链路，让列表查询真正按选中的排序方式返回。
- 在分类列表中支持拖拽排序，并把结果持久化到本地 storage。
- 处理搜索、筛选、多选、拖拽几种交互模式之间的互斥关系，避免状态残留。
- 在 uni-app 的 scroll-view 场景下，尽量保证“手柄优先”和“边拖边滚”同时成立。

这次实现覆盖的主要文件：

- src/pages/cardList/index.vue
- src/composables/useCardListInteraction.ts
- src/composables/useCardSelection.ts
- src/composables/useCardSortDrag.ts
- src/composables/useCardListView.ts
- src/services/cardService.ts
- src/utils/cardSort.ts
- src/types/card.ts
- src/pages/cardList/components/CardList.vue
- src/pages/cardList/components/CardBatchActions.vue
- src/pages/cardList/components/TransferCategoryDialog.vue

## 2. 问题定义

改造前有几类核心问题：

### 2.1 排序参数虽然加了，但查询没有真正落到服务层

列表页最初把排序选项接到了页面状态上，但服务层查询里把 cardSortConfig 混进了普通字段过滤，结果导致卡片列表被错误过滤，出现“卡片一张都没出来”的问题。

根因不是 UI 没接上，而是查询参数分层不清：

- `keyword` 属于模糊查询。
- `status`、`categoryId` 属于精确过滤。
- `cardSortConfig` 属于排序控制。

这三类参数必须在服务层分开处理。

### 2.2 拖拽排序和分页列表天然冲突

页面层的 `cardList` 不是分类下的全量数据，而是“当前已加载前 N 条”的前缀列表。这个设计本来是为了无限滚动，但它直接带来一个限制：

- 页面上看到的顺序，只能代表当前已加载片段。
- 真正持久化排序时，不能直接把当前 `cardList` 当成完整基线覆盖保存。

如果直接以页面当前顺序写回 storage，会把未加载的同分类卡片顺序弄乱。

### 2.3 多种交互模式会互相打架

列表页同时存在这些状态：

- 浏览
- 长按进入多选
- 搜索/筛选
- 拖拽排序

如果继续用多个布尔值各自管理，会出现这些典型问题：

- 正在多选时切排序，状态没清干净。
- 搜索结果页还能进入拖拽排序。
- 长按刚结束，点击事件又把页面跳进详情。

### 2.4 uni-app 里拖拽和滚动容易互相抢控制权

在 scroll-view 中做手势拖拽时，实际会同时遇到三套行为：

- 手柄本身的 touch 事件。
- scroll-view 原生滚动。
- 程序为了“贴边自动滚动”而主动改写的 `scrollTop`。

如果不区分“用户滚动”和“程序滚动”，最终就会出现：

- 拖着拖着不滚。
- 一滚就丢命中。
- 自动滚动和原生滚动来回打架。

## 3. 设计目标

这次方案收敛到四个明确目标：

1. 排序查询链路统一，页面、composable、服务层对排序含义一致。
2. 拖拽排序只在“分类列表且非搜索结果”时开放，不做跨分类排序。
3. 拖拽过程要有即时视觉反馈，但持久化必须基于分类全量顺序。
4. 页面只保留一个交互模式源，其他行为都围绕它做约束。

## 4. 方案选择与取舍

### 4.1 交互模式：从多个布尔值改为单一 interactionMode

最终采用：

- `browse`
- `select`
- `sort`

而不是继续保留多个布尔状态并行。

原因：

- 浏览、多选、拖拽在产品语义上就是互斥态，不应该允许并存。
- 单一模式源更容易做切换时的清理，例如退出多选时清空 selectedCards，退出拖拽时清空拖拽态。
- 页面上所有“先退出其他模式再做下一步”的逻辑都能统一收口。

`InteractionMode` 的实际类型定义现在放在 `src/types/card.ts`，不再让排序 composable 充当类型来源。

对应实现：

- 页面层保留 `interactionMode` 作为唯一模式源。
- `useCardSortDrag` 基于这个模式源计算 `isSortMode`、模式标题、模式说明和排序按钮状态。

### 4.2 拖拽排序的适用范围：只支持分类列表，不支持全部列表和搜索结果

这是刻意限制，不是功能缺失。

原因：

- 全部列表本身是跨分类聚合结果，而自定义排序是“分类内顺序”。
- 搜索结果页是按关键字筛出来的临时集合，不适合作为排序落库视图。
- 如果在这些场景里开放拖拽，用户看到的是局部集合，但要保存的是分类内全量顺序，心智模型会错位。

因此最终规则是：

- 只有 `categoryId` 存在且不处于搜索结果模式时，允许进入排序模式。
- 进入失败时直接给出明确原因提示。

### 4.3 持久化模型：最终采用 Move，而不是直接保存最终顺序数组

这轮中途讨论过两种保存策略。

方案 A：页面拖完后直接把最终顺序数组整体保存。

放弃原因：

- 页面里的 `cardList` 只是已加载片段，不是分类全量。
- 直接保存最终顺序数组，必须拿到分类全量且保证顺序绝对正确，否则会覆盖出错。

方案 B：拖拽过程中只记录最后一次有效移动 `Move`。

最终采用的数据结构：

```ts
type Move = {
  movedId: string;
  anchorId: string;
  position: 'before' | 'after';
};
```

采用原因：

- 页面本地重排只负责即时反馈。
- 松手时只需要把“最后一次有效移动”交给服务层。
- 服务层可以基于分类全量顺序重新计算并落库，不依赖页面是否只加载了前 N 条。

这个取舍的本质是：

- 页面负责交互反馈。
- 服务层负责基于完整数据做最终一致性保存。

### 4.4 落库策略：重写整个分类的 sort，而不是只改单张卡片的 sort

最终在 `updateCardOrderInCategory` 里采用的做法是：

1. 读取 storage 全量卡片。
2. 过滤出当前分类下的所有卡片，并按当前自定义顺序排序。
3. 应用这次 `Move`。
4. 按新的分类内顺序重写每张卡片的 `sort`，使用固定步长 `SORT_STEP = 1000`。
5. 写回整份卡片列表。

没有采用“只更新被拖动卡片的 sort 值”这类局部写法，原因是：

- 局部插值方案看起来省写入，但边界更多。
- 多次拖拽后容易出现 sort 过密、比较不稳定或需要额外整理的问题。
- 当前数据规模下，直接重写整个分类顺序更直观，也更容易复盘和排错。

## 5. 最终实现结构

### 5.1 页面层：负责参数、模式切换和普通列表行为

`src/pages/cardList/index.vue` 负责：

- 解析页面参数。
- 统一组装查询参数 `buildQuery(page)`。
- 承接搜索、状态筛选、排序切换、分页加载。
- 通过 `useCardListInteraction` 持有唯一模式源。
- 通过 `useCardSelection` 承接长按进入多选、选中切换和退出清理。
- 把拖拽排序相关逻辑委托给 `useCardSortDrag`。
- 编排 `CardList`、`CardBatchActions` 和 `TransferCategoryDialog`，不再把这些块混在同一个大模板里。

页面层保留的关键原则：

- 所有重新加载都走 `loadCurrentPages()`。
- 搜索、筛选、排序切换前先显式清理选择态，再把模式切回浏览态。
- 页面只关心“当前是什么模式”，不直接处理拖拽命中和自动滚动细节。

### 5.2 useCardListView：负责“当前已加载前缀列表”

`src/composables/useCardListView.ts` 的设计没有改成真正的分页页码追加，而是继续维持当前项目已经在用的“请求前 N 条”策略：

- 第 1 页取前 10 条。
- 第 2 页取前 20 条。
- 第 3 页取前 30 条。

这能保持当前列表视图和返回页面后的数据体验一致，但也带来一个明确前提：

- `cardList` 只是已加载前缀，不是全量基线。

所以排序保存不能依赖它。

### 5.3 useCardSortDrag：负责拖拽会话、命中计算和滚动协调

`src/composables/useCardSortDrag.ts` 承担的职责有：

- 排序模式是否允许进入。
- 排序模式的 UI 文案。
- 当前拖拽态卡片的激活状态。
- 卡片 rect 和 scroll-view rect 的采集。
- 手指命中的锚点计算。
- 本地列表即时重排。
- 贴边自动滚动。
- touchend 时把最后一次 move 交给服务层保存。

拖拽链路的实际顺序是：

1. `touchstart` 时记录当前拖拽卡片，并采集一次 rect。
2. `touchmove` 时先判断是否需要自动滚动。
3. 再根据最新 rect 计算当前命中的锚点卡片和落点位置。
4. 命中有效目标后，本地调整 `cardList` 顺序并刷新 rect。
5. `touchend` 时清理临时状态，并将最后一次 `Move` 提交给服务层。

### 5.4 cardService：负责最终一致性保存

`src/services/cardService.ts` 在这轮里新增了两个关键点：

- `getCards` 中把 `cardSortConfig` 从过滤字段里拆出来，单独作为排序配置处理。
- `updateCardOrderInCategory(categoryId, move)` 负责按分类全量顺序应用拖拽结果并保存。

这里的责任边界很重要：

- 页面不做最终顺序真值。
- 服务层才是顺序持久化的唯一落点。

## 6. 关键实现细节

### 6.1 为什么进入排序模式时强制切到 customSort

如果当前列表还停留在“按创建时间”或“按更新时间”排序，用户看到的顺序就不是自定义顺序。此时允许拖拽，会出现明显的认知冲突：

- 拖的是一套顺序。
- 保存的却是另一套自定义顺序。

因此进入排序模式时，会先把排序项切到 `customSort`，再展示拖拽手柄。

退出排序模式时不自动切回原排序，原因也明确：

- 用户刚刚排完序，立即切回别的排序会让列表看起来“排序没生效”。

### 6.2 为什么本地重排后还要频繁 refreshCardRects

因为每次局部重排或滚动后，屏幕上每张卡片的位置都变了。命中逻辑依赖 rect，如果不重新采集，下一次 touchmove 很容易按旧坐标继续算，表现出来就是：

- 命中错卡。
- 拖动时顺序跳变。
- 到边缘后拖拽目标突然失准。

APP 端还额外要求：rect 不能只按数组下标和 `cardList` 对齐。拖拽时会同时出现本地重排、透明源节点、占位卡、分页追加，所以下标很容易失真。当前设计要求卡片节点带 `data-card-id`，rect 采集时读取 `dataset`，命中前按屏幕 `top` 排序，避免新页节点或 APP 返回顺序导致错位。

分页追加也属于 rect 必须重建的场景。拖拽中允许触发分页加载，但加载完成后必须立刻重采 rect，并用当前触点刷新占位；不能继续拿旧页 rect 命中新页。

### 6.3 APP 拖拽视觉层：源节点保留，代理卡显示

H5 端可以把拖拽源卡片从列表里隐藏，再用代理卡显示浮起状态。但 APP 端不能简单移除或 `display: none` 源节点，因为原生触摸链路可能绑定在最初按住的节点上。源节点一旦消失，后续 `touchmove` 可能中断。

因此 APP 端采用：

- 源卡片保留在 DOM 中，通过 `is-app-detached-drag-source` 透明并移出文档流。
- 用户看到的浮起卡片由 `CardDragProxy.vue` 渲染。
- APP 列表不使用 `transition-group`，改用普通 `view + v-for`，降低节点替换导致的触摸中断风险。

这个设计的责任分离是：

- 源节点负责维持触摸事件。
- 代理卡负责视觉跟手。
- 占位卡负责表达最终落点。

### 6.4 占位卡下标为什么是显式状态

早期 `sortInsertIndex` 由 computed 每帧根据当前触点和 rect 现场计算。APP 真机长列表里，这会出现两个问题：

- 某一帧命中失败，computed 返回 `-1`，占位卡瞬间消失。
- 某一路径只更新了 `sortDragPreviewMove`，没有同步更新下标，导致占位卡停在旧位置。

当前设计改为显式状态：只有 `applySortDragTarget` 拿到有效锚点时，才同时更新 `sortDragPreviewMove` 和 `sortInsertIndex`。命中失败、命中自己、锚点暂时不可用时，不主动清空占位，而是保留上一帧有效落点。

### 6.5 尾部落点为什么单独处理

尾部和顶部看起来对称，但交互上不对称：

- 顶部落点只要进入第一张非源卡的上半区，就能表达 `before`。
- 尾部落点通常需要越过最后一张的下半区或底边，才能表达 `after`。
- APP 底部固定操作栏不占列表布局，却会盖住最后一张卡的自然 after 区域。
- 当拖拽源卡片本身是最后一张时，命中列表排除源卡，尾部锚点容易退化成倒数第二张。

因此当前尾部规则是：下拖时，如果命中点进入最后一个可见锚点的下半区，直接判定为插入当前已加载列表尾部；尾部 fallback 使用当前已加载列表里的最后一张非源卡，而不是最后一个 rect。

### 6.6 如何区分原生滚动和程序自动滚动

这部分是这轮里最容易踩坑的实现点。

当前方案使用这些状态协同：

- `lastAutoScrollAt`：最近一次程序自动滚动的时间。
- `programmaticScrollUntil`：代码修改 `scrollTop` 后的回响保护窗口。
- `scrollTop`：由 `@scroll` 记录真实滚动位置。
- `scrollTopTarget`：只作为程序推进 `scroll-view` 的目标值。

工作方式：

1. 程序推进时只写 `scrollTopTarget`，不直接把真实 `scrollTop` 当目标值混用。
2. `@scroll` 回来后只更新真实 `scrollTop`。
3. `watch(scrollTop)` 先用滚动差值平移缓存 rect，再安排真实 rect 重采。
4. 自动滚动循环每推进一次视口，都用当前触点刷新一次占位。

这样做的目的是避免 APP 端 `@scroll` 回写吞掉程序滚动目标，同时保证“视口在滚”和“落点在更新”不脱节。

## 7. 后续收口补充

这份复盘最初只记录了“排序拖拽如何工作”，但后来卡片列表页又继续做了结构收口，所以这里补一层，避免后续只看这份文档时误以为多选和批量操作还绑在页面里。

### 7.1 交互模式的类型来源被统一

`InteractionMode` 最终保留在 `src/types/card.ts`。这样做的原因是：

- 它本质是卡片列表交互的公共语义，不属于排序 composable 独占。
- 多选逻辑、排序逻辑、页面模式切换都要共用这个类型。
- 类型放在共享模型层，比放在某个具体行为 composable 里更稳定。

### 7.2 多选逻辑从页面里拆出去，但退出语义收口到页面

后续为了让页面主文件更轻，多选逻辑被挪到 `useCardSelection`，但保留了一个关键原则：

- `useCardSelection` 负责清理选择态和触摸态。
- 页面负责决定什么时候切回浏览态。

这样做的好处是语义清晰：

- “退出多选”只表示把选择相关状态清掉。
- “回到浏览态”是页面级决策，不会被子逻辑偷偷完成。

### 7.3 列表、批量操作和弹窗被拆成独立 UI 块

后续列表展示又进一步拆成：

- `CardList`：只负责列表视觉和事件。
- `CardBatchActions`：只负责批量操作栏。
- `TransferCategoryDialog`：只负责分类转移弹窗。

这一步不是为了继续切碎，而是为了让“排序拖拽设计”不再和“批量操作 UI”纠缠在同一个模板里。

## 8. 踩坑记录

### 8.1 排序参数误参与精确过滤，直接导致空列表

这是最先暴露出来的问题。现象是“排序接上后卡片一张都没出来”，根因是：

- `getCards(params)` 里把 `cardSortConfig` 当普通字段参与了卡片过滤。

修复方式：

- 在服务层先把 `cardSortConfig` 解构出去，只对剩余字段做精确匹配。

### 8.2 把页面局部列表误当成分类全量顺序

这是排序持久化里最大的结构性风险。

最开始如果沿着“页面已经排好了，直接保存页面顺序”这个思路走，会天然踩到分页前缀列表的坑。后面明确改成 `Move + 服务层全量回放`，就是为了解这个问题。

### 8.3 文案和真实行为不一致

中途出现过“完成排序”这类文案，但实际行为并不是“点击后统一提交排序”，而是：

- 拖拽时本地跟手。
- 松手时自动保存。
- 按钮本质是进入/退出排序模式。

所以后来把文案改成“拖拽排序 / 退出拖拽”，避免 UI 暗示错误的交互模型。

### 8.4 粗暴禁用滚动并不能解决拖拽问题

一开始很容易走到一个表面正确的方案：

- 进入拖拽后把原生滚动关掉。

但这样会直接带来另一个问题：

- 用户拖到长列表边缘时，根本没法继续往下拖。

最终还是回到“保留原生滚动，但用静默窗口协调程序滚动”的方案。

### 8.5 手柄事件必须优先接管

如果不在手柄上使用 `touchstart/touchmove/touchend` 的 `stop + prevent`，拖拽过程中父级卡片的点击、长按、多选逻辑就很容易混进来，导致：

- 拖着拖着触发多选。
- 松手后误进详情。

所以最终明确采用“点中手柄，优先手柄”的事件策略。

### 8.6 顶边上拖为什么会“鬼打墙”

这次排查里最关键的一个问题是：拖到列表顶边时，程序自动滚动一旦触发，就会把命中和本地重排拆成两个互相抢时机的阶段。

最初的实现里，`runSortDragMoveFrame` 在检测到自动滚动后直接早退，结果是：

- 这一帧只滚动视口，不再继续算 `getDragTargetByTouchY`。
- `moveCardLocally` 不会执行，列表顺序不会立刻变化。
- 占位卡在边缘附近没有新的落点，只能短暂依赖上一帧状态。

这就是上拖时最容易感知到的“卡住、不往上推、像在原地打转”。

后续修正保留了自动滚动，但不再让它阻断同一帧里的命中计算和本地重排。这样做的目标不是让滚动更激进，而是让“视口在滚、卡片在换位”这两件事继续同步。

这次更细的排障过程和验证步骤单独收在 [./card-list-sort-drag-debug.md](./card-list-sort-drag-debug.md) 里，方便后续直接按现象回看根因。若要看从“手感改造”到“顶边死锁修复”的完整过程，去看 [./card-list-sort-drag-full-record.md](./card-list-sort-drag-full-record.md)。

## 9. 这次方案的边界

当前实现明确不处理这些能力：

- 跨分类拖拽排序。
- 搜索结果集合内拖拽排序。
- 全部列表视图下的拖拽排序。
- 基于页面当前局部列表直接覆盖最终顺序。

这些不是遗漏，而是当前数据模型和交互模型下的主动约束。

## 10. 验证情况

这轮改动完成后，已做过的校验包括：

- 相关核心文件的文件级错误检查通过。
- 关键控制链已补注释，便于后续继续排查和迭代。

当前仍保留的验证边界：

- 项目全量 `type-check` 仍受 tsconfig 中 `ignoreDeprecations: "6.0"` 影响，现阶段无法作为有效收口校验。
- 拖拽手感参数（如边缘触发距离、滚动步长、静默窗口）仍然是体验参数，需要继续在真机和长列表场景里微调。
